# Phase 02 Naming And Taxonomy Implementation

**Status:** Research complete, ready for implementation  
**Date:** 2026-03-07

## Objective

Define one canonical user-facing taxonomy for knowledge, scope, and session
copy so the project-first UI stops leaking raw enum keys, internal field names,
and generator jargon.

This phase is complete when there is a single rename matrix for display terms
versus wire terms, the web app can render those labels from one shared
taxonomy layer, and future schema cleanups are optional migrations rather than
required to fix the UX.

## Non-Goals

- rewrite autogenerated component instance names like `Authentication Module`
  or `Unknown Module` in this phase
- redesign the knowledge filter bar interaction model
- redesign the session lobby or event rendering model
- introduce a new project model beyond the Phase 0 ownership contract
- change the Phase 1 route map or navigation shell
- break persisted blueprint or session payloads purely to improve copy
- adopt broad domain metaphors like `Workstream` everywhere without explicit
  product confirmation

## Decision Summary

- Keep raw enum values and persisted field names as internal contract terms in
  the first rollout.
- Add one shared taxonomy registry in the web app as the only source for:
  - singular labels
  - plural labels
  - short labels for compact badges
  - field labels
  - helper text for forms and filters
- Standardize `quality_requirement` as `Quality Scenario` in normal UI copy.
  A short label `Quality` is acceptable only in dense badges or charts.
- Simplify scope taxonomy:
  - `Scope Class` -> `Placement`
  - `Scope Visibility` -> `Availability`
  - `project_contextual` -> `Project Context`
  - `project_local` -> `Project Only`
  - `unscoped` stays `Unscoped` in neutral tables and filters, but becomes
    `Needs Scope` in intervention and health contexts
- Simplify component subtype taxonomy:
  - `module` -> `Application Module`
  - `store` -> `Data Store`
  - `interface` -> `Interface Surface`
  - `pipeline` -> `Automation Pipeline`
- Rewrite shared-scope copy around sharing behavior instead of storage
  mechanics:
  - `Shared knowledge` -> `Shared across projects`
  - `Linked Project IDs` -> `Shared with projects`
  - `Overrides Shared Source ID` -> `Overrides shared record`
  - `inherit_to_linked_projects` -> `Show in linked projects`
- Relabel `Session.project_description` in the UI as `Planning brief`. The
  stored field name remains unchanged until a later compatibility migration is
  explicitly approved.
- Use the existing session status pattern in `Dashboard.tsx` as the model for
  the rest of the app: internal keys stay internal, user-facing surfaces render
  deliberate copy.

## Current-State Summary

There is no single naming system today. The app duplicates label maps across
pages, components, forms, and schema helpers, so the same underlying value can
appear under multiple names depending on where the user lands.

| Concept | Internal key | Current labels in code | Current problem |
| --- | --- | --- | --- |
| Quality node type | `quality_requirement` | `Quality`, `Quality Req.`, `Quality Requirement`, `QualityRequirement` | one type has four different labels |
| Project-context scope | `project_contextual` | `Project Contextual` | schema jargon leaks directly into the UI |
| Scope visibility | `project_local` | `Project only`, `Project Local` | same state has inconsistent copy |
| Component subtype | `module`, `store` | `Module`, `Store`, `Data Store` | implementation nouns are shown as product taxonomy |
| Session brief | `project_description` | `project description`, `saved description` | conflicts with the Phase 0 canonical `Project` model |
| Sharing controls | `linked_project_ids`, `shared_source_id` | `Linked Project IDs`, `Overrides Shared Source ID` | storage-oriented wording appears in user forms |

### Current code anchors

- `KnowledgeLibraryPage.tsx` defines inline node-type labels, scope filters,
  visibility filters, and intervention copy.
- `NodeListPanel.tsx`, `DetailDrawer.tsx`, `TableView.tsx`, and
  `BlueprintPage.tsx` each define their own type or scope label maps.
- `AddEdgeModal.tsx` still renders `QualityRequirement` without spacing inside
  relation descriptions.
- `CreateNodeModal.tsx` and `EditNodeForm.tsx` expose raw field labels such as
  `Scope Class`, `Project Contextual`, `Project ID`, `Linked Project IDs`, and
  `Overrides Shared Source ID`.
- `planner-schemas/src/artifacts/blueprint.rs` already normalizes some status
  strings for display, but node types, scope terms, and component subtypes
  still rely on scattered frontend copy.
- `Dashboard.tsx` and `SessionPage.tsx` prove the internal-to-display mapping
  pattern works for workflow states, but the saved input is still described as
  a `project description`.

### Visual audit findings from 2026-03-07

- `/knowledge/all` currently exposes raw or inconsistent taxonomy in the first
  screenful of controls: `Project Contextual`, `Project only`, `Unscoped`,
  `Quality`, `Widget`, and `Component` all appear as primary filter language.
- The dense chip layout amplifies terminology drift because the same concepts
  repeat across many rows. Even acceptable short labels such as `Quality`
  become ambiguous when they are not backed by one shared taxonomy source.
- Some current product-language surfaces already read well, such as
  `Needs Attention` and `At Risk` on project cards. Phase 2 should preserve
  that deliberate copy and focus the cleanup on enum-driven filters, tables,
  forms, and drawers.

## Proposed Behavior

### Taxonomy architecture

Create a shared taxonomy module in the web app, for example
`planner-web/src/lib/taxonomy.ts`, with a small registry per category:

```ts
interface TaxonomyEntry<K extends string> {
  key: K;
  label: string;
  pluralLabel?: string;
  shortLabel?: string;
  fieldLabel?: string;
  helpText?: string;
}
```

Required helpers:

- `labelNodeType(key, variant?)`
- `labelScopeClass(key, context?)`
- `labelScopeVisibility(key, variant?)`
- `labelComponentType(key, variant?)`
- `labelSecondaryScopeField(key)`
- `labelSessionField(key)`

No page or component should keep its own hard-coded label map after this phase
ships.

### Terms that should stay unchanged in Phase 2

The following status families are already readable enough and should remain
semantically unchanged in this phase:

- decision status
- adoption ring
- component status
- quality priority
- lifecycle status

The implementation work here is to centralize their label source, not to invent
new words for them.

### Alias-only rollout versus contract cleanup

Phase 2 should deliberately split naming work into two tracks.

#### Track A: UI alias rollout

- Replace scattered label maps with shared helpers.
- Update visible copy in filters, forms, tables, drawers, badges, and session
  screens.
- Keep all current JSON keys, query params, and persisted enum values.

#### Track B: future contract cleanup candidates

Only start this if the product explicitly wants API or schema terminology to
change after the UI layer has stabilized.

Candidate future raw-key renames:

- `quality_requirement` -> `quality_scenario`
- `project_contextual` -> `project_context`
- `project_local` -> `project_only`
- `project_description` -> `planning_brief`

These are not required to complete the Phase 2 UX fix.

## API And Data Model Changes

Phase 2 should start with a non-breaking contract.

### First rollout

- Keep existing JSON keys such as `node_type`, `scope_class`,
  `scope_visibility`, `component_type`, and `project_description`.
- Keep existing query params in `planner-web/src/api/client.ts` such as:
  - `type`
  - `scope_class`
  - `scope_visibility`
  - `project_id`
- Do not rewrite persisted blueprint nodes or session MessagePack records in
  the UI-alias rollout.

### Shared schema helpers

If server-generated exports, summaries, or event payloads need the same
terminology, add helper methods in `planner-schemas` for display labels rather
than duplicating more string maps in the server.

That helper layer must remain non-authoritative for persistence. Raw enums stay
the system of record.

### Optional future compatibility window

If the product later wants contract cleanup:

- the server must accept both old and new enum values during a compatibility
  window
- incoming aliases must normalize to one internal representation at the API
  boundary
- tests and fixtures must cover both old and new spellings before any data
  rewrite begins

No such migration is required to complete the Phase 2 implementation document.

## UI And Routing Changes

No route-map changes are required for Phase 2. This is a taxonomy and copy
cleanup phase.

### Knowledge and blueprint surfaces

- Replace all local `NODE_TYPE_LABELS`, `TYPE_LABELS`,
  `SCOPE_CLASS_LABELS`, and `SCOPE_VISIBILITY_LABELS` tables with shared
  taxonomy helpers.
- Use long labels in filters, headings, empty states, and drawer section
  headings.
- Use short labels only in dense badges, chart legends, or cramped tables.
- Eliminate these inconsistent variants:
  - `Quality`
  - `Quality Req.`
  - `Quality Requirement`
  - `QualityRequirement`
  - `Project Contextual`
  - `Project Local`
- Rename health and intervention copy from `Unscoped records` to `Needs scope`
  where the UI is asking the user to take action.

### Create and edit forms

- `Scope Class` becomes `Placement`.
- `Scope Visibility` becomes `Availability` wherever it is shown.
- `Project Contextual` becomes `Project Context`.
- `Feature` becomes `Feature Area`.
- `Widget` becomes `Surface`.
- `Shared knowledge` becomes `Shared across projects`.
- `Linked Project IDs` becomes `Shared with projects`.
- `Overrides Shared Source ID` becomes `Overrides shared record`.
- `Quality Requirement` becomes `Quality Scenario` in node-type selectors.
- Component subtype selectors use the renamed subtype labels from the matrix.

### Session surfaces

- `project description` becomes `planning brief`.
- `saved description` becomes `saved brief`.
- `initial project description` becomes `planning brief`.
- Empty-state and restart copy must stop implying that the free-text session
  prompt is the same thing as a canonical project identity.

### Project reference wording

Until Phase 0 project selection exists in the UI, raw project reference fields
may remain technically backed by `project_id`, but they should be treated as
advanced scope inputs rather than normal product language.

Once project selection exists, users should pick projects by name or slug
instead of typing IDs manually.

## Migration And Backfill Plan

### Step 1: Add the taxonomy registry

- create the shared taxonomy module
- add tests for all canonical labels and variants
- move each web surface to taxonomy helpers instead of local constants

### Step 2: Update session and form copy

- replace `project description` wording with `planning brief`
- update form labels for scope, sharing, and subtype selection
- keep wire field names unchanged

### Step 3: Update tests and docs

- change UI assertions to the new labels
- update implementation docs, README references, and screenshots if they
  include renamed copy

### Step 4: Evaluate contract cleanup only if still needed

- add server-side alias acceptance for new raw values
- normalize aliases at the API boundary
- update fixtures and integration tests before any stored-data rewrite

### Step 5: Optional persisted cleanup

Only if contract cleanup is approved later:

- backfill stored blueprint nodes and session records to the chosen new keys
- keep import and export compatibility for older payloads during the migration
  window

### Migration safety rules

- never mix old and new raw enum values within the same persisted domain object
- normalize aliases before validation and persistence
- do not rewrite historical data until the compatibility path is fully tested
- do not block the UI label cleanup on a future raw-key migration

## Tests To Add Or Update

## Risks, Dependencies, And Rollout Order

### Main risks

- taxonomy drift will continue if even one major surface keeps a local label
  map
- raw enum renames would break persisted blueprint nodes, API validation, and
  fixtures if started too early
- relabeling subtype taxonomy before Phase 3 may make generic autogenerated
  names feel more obviously weak
- project reference wording cannot be fully fixed until the Phase 0 project
  selector exists

### Dependencies

- Phase 0 project ownership work for eventual project-picking UI and the final
  semantics of `Planning brief`
- Phase 1 shell work if shared headers or nav copy end up reusing taxonomy
  helpers
- Phase 3 should follow soon after so generic component names stop undercutting
  the cleaner subtype taxonomy

### Recommended rollout order

1. Land the shared web taxonomy registry and remove duplicated label maps.
2. Update knowledge and blueprint surfaces.
3. Update create/edit forms and session copy.
4. Update tests and docs.
5. Evaluate API or schema alias support only if the UI-only rollout proves
   insufficient.

## Unresolved Questions

- Should the long-form label remain `Quality Scenario`, or would the product
  prefer `Quality Requirement` once users react to the new terminology?
- Should `widget` become `Surface` or `View`?
- Once Phase 0 projects exist in the UI, should raw project-reference fields
  disappear entirely instead of merely being relabeled as advanced inputs?
- Does the product want eventual API/schema cleanup, or is a permanent UI alias
  layer acceptable?

## Done When

Phase 2 is complete when the following statements are true:

- every user-visible blueprint and session taxonomy label is sourced from one
  shared taxonomy module
- `quality_requirement`, `project_contextual`, and `project_local` no longer
  appear in UI copy
- `project_description` no longer appears in user-facing session copy
- filters, forms, tables, drawers, badges, and session screens use the agreed
  labels consistently
- tests guard against taxonomy drift and any future contract cleanup is clearly
  separated from the UI alias rollout
