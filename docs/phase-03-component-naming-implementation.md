# Phase 03 Component Naming Implementation

**Status:** Research complete, ready for implementation  
**Date:** 2026-03-07

## Objective

Replace weak autogenerated component names with deterministic, useful names
that help users distinguish system parts from the label alone, while ensuring
manual component renames survive future generator and discovery reruns.

This phase is complete when new and regenerated component nodes no longer fall
back to labels like `Authentication Module`, `001 Module`, `Api`, `Core`, or
`Factory Output`, and the system can tell whether a component name is
generated or manually owned.

## Non-Goals

- rename non-component node types such as decisions, technologies, patterns,
  constraints, or quality scenarios
- redesign the global taxonomy labels already defined in Phase 02
- change the Phase 01 route model or navigation shell
- introduce an LLM-driven naming service for component names
- fully redesign the discovery workflow beyond the naming and accept flows
- solve every duplicate-node problem in the blueprint model for all node types
- remove the current `NodeId` format or break persisted blueprint payloads

## Decision Summary

- Add one shared component naming engine in `planner-core` and stop generating
  component names from ad hoc helper functions in separate modules.
- Treat component instance names and component subtype taxonomy as different
  concerns:
  - the `name` should describe the capability or surface
  - `component_type` should keep classifying the node as `service`, `store`,
    `interface`, `pipeline`, and so on
  - the system must never add `Module` as the default suffix again
- Extend `Component` nodes with optional naming provenance metadata so the
  system can tell whether a name is generated or manual.
- Add a stable `origin_key` for generated components and use it for lookup,
  dedupe, and regeneration instead of using the current display name.
- Manual names always win:
  - a manual rename changes naming ownership to `manual`
  - later generator or discovery passes may update the latest suggestion, but
    they must not overwrite the active `name`
- Replace name-based discovery dedupe with origin-based dedupe so a better
  naming algorithm does not create duplicate proposals for the same source.
- Add a safe backfill pass for obviously generic legacy names, but only
  auto-rename nodes that are clearly generated and not manually curated.

## Current-State Summary

Today component names come from several unrelated paths, each with different
heuristics and no shared provenance model.

| Surface | Current behavior | Current problem |
| --- | --- | --- |
| NLSpec compile emission | `emit_from_spec` groups requirements by ID segment and calls `humanize_group_name` | names depend on coarse group tokens and fall back to `X Module` |
| Requirement group fallback | `FR-001` becomes group `001`, then `001 Module`; empty groups can become `Unknown Module` | numeric or malformed groups leak directly into the UI |
| Hard-coded group aliases | `auth` -> `Authentication Module`, `api` -> `API Layer`, `ui` -> `User Interface`, `payment` -> `Payment System` | role words are inconsistent, subtype jargon leaks into names, and synonyms are not normalized |
| Factory emission | stage 4 always creates a component named `Factory Output` | the name is generic and does not identify the project or artifact |
| Directory discovery | `humanize_name` title-cases the last path segment only, so `api`, `core`, and `src/web` become `Api`, `Core`, and `Web` | proposed names are often too short to identify system responsibility |
| Discovery dedupe | existing nodes and proposals are deduped by lowercased name | improving the naming algorithm can create duplicates for the same artifact |
| Manual editing | the web app lets users rename component nodes, but `Component` has only a plain `name` field | future regeneration cannot tell whether the current name is user-owned or system-generated |

### Code findings that make regeneration unsafe today

- `Component` nodes persist only `name`, `component_type`, and descriptive
  fields. There is no `name_source`, `generated_name`, or `origin_key`.
- `emit_from_spec` creates new component IDs from group names but does not
  persist a stable generator identity that can be reused on later runs.
- `scan_directory_structure` dedupes against existing component names, not
  source paths, and proposal fingerprints also include the proposed name.
- `accept_proposal` merges the proposal node directly into the blueprint, so
  any discovery source path or naming rationale is lost from the node itself.
- The blueprint event log stores `before` and `after` on updates, which is
  enough to support rollback and migration review, but it does not solve
  naming provenance on its own.

### Visual audit findings from 2026-03-07

- The live Blueprint graph currently renders multiple nodes as `1 Module`,
  `2 Module`, `4 Module`, `5 Module`, and `7 Module`. The graph is visually
  scannable only after opening details or inferring intent from nearby nodes.
- The `/knowledge/all` table repeats the same weak labels in the primary
  `Name` column, so the naming problem affects both spatial graph views and
  dense audit tables.
- These weak names visually overpower more useful metadata such as type,
  status, and scope badges because the user must first decode whether the row
  is worth opening at all. Phase 3 is therefore a readability fix, not just a
  generator cleanup.

## Proposed Behavior

### Naming architecture

Create a shared core module, for example
`planner-core/src/component_naming.rs`, that exposes one deterministic naming
entry point for all generated component paths.

Suggested shape:

```rust
pub enum ComponentNameSource {
    Generated,
    Manual,
}

pub enum ComponentNamingStrategy {
    SpecGroup,
    DirectoryScan,
    FactoryOutput,
    ManualCreate,
    Backfill,
}

pub struct ComponentNaming {
    pub origin_key: String,
    pub source: ComponentNameSource,
    pub strategy: ComponentNamingStrategy,
    pub generated_name: String,
    pub naming_version: u16,
    pub last_generated_at: String,
}
```

Add the metadata as an optional field on `Component`:

```rust
pub struct Component {
    pub id: NodeId,
    pub name: String,
    pub component_type: ComponentType,
    pub naming: Option<ComponentNaming>,
    ...
}
```

The system of record for rendering remains `component.name`.

The purpose of `component.naming` is to answer four questions:

- What generator or source produced this name?
- What stable source identity should future passes match against?
- What is the latest generated suggestion?
- Is the active name still system-owned or now manual?

### Unified naming rules

The naming engine should build names in two stages:

1. Resolve a subject.
2. Resolve an optional role suffix.

Formatting rule:

- Preferred output is `<Subject>` or `<Subject> <Role>`.
- Keep names short and specific:
  - usually 1-4 words
  - preserve known acronyms such as `API`, `UI`, `JWT`, `OAuth`
- Never append `Module` as the default suffix.
- Avoid duplicate role words such as `API Service API` or `Admin UI Interface`.

### Subject extraction rules

Use the following precedence order:

1. Strong alias map for common requirement and path tokens:
   - `auth` -> `Authentication`
   - `billing`, `payment`, `payments` -> `Billing`
   - `notify`, `notification`, `notifications` -> `Notifications`
   - `search` -> `Search`
   - `admin` -> `Administration`
   - `config`, `settings` -> `Configuration`
   - `sync` -> `Sync`
   - `report`, `reporting` -> `Reporting`
   - `analytics` -> `Analytics`
2. Structured source context:
   - requirement statements
   - directory parent segments
   - project feature or chunk name
   - output path basename
3. Normalized path or group tokens after removing boilerplate such as:
   - `src`
   - `crates`
   - `packages`
   - `app`
   - `lib`
   - `general`
4. Project name plus the best surviving token if the source is otherwise too
   generic.

If the source token is purely numeric or generic, such as `001`, `api`, `ui`,
`core`, or `web`, the naming engine must not use that token alone as the final
subject.

### Role selection rules

Role suffixes are optional and should come from evidence, not from fallback
jargon.

Allowed role suffixes:

- `Service`
- `API`
- `UI`
- `Worker`
- `Data Store`
- `Pipeline`
- `Library`
- `Gateway`

Role hints should come from:

- `component_type`
- requirement text
- directory path clues
- known aliases

Examples:

| Source signal | Preferred output |
| --- | --- |
| auth requirements about login, JWT, sessions | `Authentication Service` |
| endpoint or schema requirements for HTTP or GraphQL | `Public API` or `Admin API` |
| UI path or view requirements | `Dashboard UI`, `Project Hub UI` |
| store, db, cache paths | `Session Data Store`, `Knowledge Cache` |
| worker, sync, import, export paths | `Task Sync Worker`, `Export Pipeline` |

If the role is not confidently supported, omit it and keep a subject-only name
instead of falling back to `Module`.

### Weak-name blacklist

The naming engine must reject or backfill these categories of outputs:

- `Unknown Module`
- any name that ends with ` Module`
- single-token placeholders such as `Api`, `Core`, `Web`, `Lib`, or `Store`
- `Factory Output`
- raw numeric labels such as `001 Module` or `001`

### Source-specific generation behavior

#### NLSpec component groups

Replace `humanize_group_name` with the shared naming engine.

Generation input should include:

- normalized requirement group token
- all statements in the group
- `project_id`
- chunk name or feature name when available
- inferred `component_type`

The phase must also stop treating the group token as the only identity.
Instead, create a stable `origin_key` such as:

- `spec:{project_id}:{chunk_tag}:{group_token}`

On later runs:

- if a component with the same `origin_key` exists and `naming.source` is
  `generated`, update the node in place
- if a component with the same `origin_key` exists and `naming.source` is
  `manual`, update non-name fields but leave `name` untouched
- if no component with the `origin_key` exists, create a new node

This is the minimal behavior needed to make regeneration compatible with manual
renames without changing public node IDs.

#### Directory discovery

Replace `humanize_name` with the shared naming engine.

Generation input should include:

- full relative path
- normalized path segments
- inferred `component_type`
- project root context

Use an `origin_key` such as:

- `path:{relative_path}`

Name construction rules:

- ignore boilerplate segments like `src`
- combine parent and child segments when the leaf is generic
- use the inferred role only when it adds meaning

Examples:

| Relative path | Current output | New target behavior |
| --- | --- | --- |
| `api` | `Api` | `API Service` |
| `src/web` | `Web` | `Web App` or project-specific `Project Hub UI` if project context exists |
| `services/task_sync` | `Task Sync` | `Task Sync Service` |
| `admin/api` | `Api` | `Admin API` |

#### Factory output

The phase should stop creating a cross-project generic component named
`Factory Output`.

First-rollout behavior:

- keep the synthetic node if build-status visibility still needs it
- derive its name from project and output context, for example
  `<Project> Generated Workspace` or `<Output Basename> Workspace`
- persist `origin_key = factory:{output_path}`

Longer-term note:

- if a later phase decides that generated output is build evidence rather than
  a domain component, that node can be demoted or removed separately

#### Manual creation and manual rename

Manual component creation from `CreateNodeModal` should set:

- `naming.source = manual`
- `naming.strategy = manual_create`
- `generated_name = current name`
- `origin_key = manual:{node_id}`

Manual rename behavior in `EditNodeForm` and API patch handling:

- if a component name changes to a different non-empty value through a user
  edit, set `naming.source = manual`
- keep the latest generated suggestion in `generated_name` if it exists
- do not discard `origin_key`

### Manual names versus regeneration

The overwrite rules should be explicit:

| Current state | Generator rerun behavior |
| --- | --- |
| `source = generated` and `name == generated_name` | update `name` and `generated_name` |
| `source = generated` and current name matches a weak-name fallback | update `name` and `generated_name` |
| `source = manual` | update `generated_name` only; keep `name` |
| no naming metadata, but name is obviously generic and source is generated | backfill metadata and allow safe rename |
| no naming metadata, name is specific, and source is ambiguous | preserve current `name`, record suggestion only |

This phase should not silently overwrite names that look hand-curated.

## API And Data Model Changes

### Schema changes

Add optional naming metadata to `Component` on the Rust and TypeScript sides.

This is a non-breaking additive change.

Required fields inside the optional `naming` object:

- `origin_key`
- `source`
- `strategy`
- `generated_name`
- `naming_version`
- `last_generated_at`

No other node type needs the new metadata in Phase 03.

### Server behavior changes

`PATCH /blueprint/nodes/{nodeId}` must gain component-specific normalization:

- if a component name changes and the request does not explicitly declare a new
  naming source, coerce `naming.source` to `manual`
- retain `origin_key`
- update `updated_at` as normal

### Discovery accept flow

Change `POST /blueprint/discovery/proposals/{id}/accept` so it can optionally
accept a node patch for final review-time edits:

```json
{
  "node_patch": {
    "name": "Task Sync Service",
    "naming": {
      "source": "manual"
    }
  }
}
```

This avoids forcing the user to accept a weak generated name first and rename
it second.

### Dedupe and lookup changes

Generated component lookup must stop relying on the current display name.

Required behavior changes:

- discovery proposal fingerprinting should use `origin_key` instead of
  `node.name()`
- existing generated component lookup should use `origin_key`
- generator reruns should update by `origin_key`, not create a new node just
  because the generated name improved

### Node summary payloads

No summary-payload change is required in the first rollout.

The list views can continue consuming `name`. Full node payloads are enough for
the drawer and edit flows that need provenance detail.

## UI And Routing Changes

No route changes are required for Phase 03.

### Discovery page

For component proposals:

- show the suggested name prominently
- allow inline rename before acceptance
- show whether the accepted result will remain generated or become manual
- keep showing the source artifact and rationale so the user understands why
  the name was suggested

### Create and edit flows

`CreateNodeModal`:

- manual component creation should initialize naming provenance as manual
- the label remains `Component name`; subtype selection remains separate

`EditNodeForm` and `DetailDrawer`:

- show `Name source: Generated` or `Name source: Manual`
- when a generated suggestion differs from the current name, show the latest
  suggestion
- add a `Use suggested name` action for components whose active name is manual
  but whose generated suggestion changed
- explain that manual names are preserved across regeneration

### Blueprint and knowledge surfaces

The rest of the UI should continue rendering `component.name`.

Phase 03 should not require custom name formatting in list pages. The work is
to improve the stored display name and expose provenance only where editing or
review happens.

## Migration And Backfill Plan

### Step 1: Add naming provenance and the unified engine

- add the new component naming metadata to the schema
- build the shared core naming module
- route spec emission, discovery scanning, and factory emission through it

### Step 2: Add stable origin lookup

- persist `origin_key` on every generated component
- add helper lookup in blueprint storage to find a component by `origin_key`
- switch discovery proposal dedupe from name-based to origin-based matching

### Step 3: Add manual-ownership rules

- make manual creation and manual rename mark `naming.source = manual`
- keep `generated_name` as the latest system suggestion
- ensure generator reruns update only generated-owned names

### Step 4: Produce a backfill preview

Before applying any rename, create a blueprint history snapshot using the
existing history mechanism.

Then run a preview-only migration that reports:

- `node_id`
- current name
- suggested name
- inferred origin
- confidence
- proposed action:
  - `auto_rename`
  - `suggest_only`
  - `skip`

### Step 5: Auto-rename only safe legacy nodes

Safe auto-rename candidates:

- components tagged from generated sources such as `spec`, `discovery`, or
  `factory`
- names that match the weak-name blacklist
- names that still equal their known generated value

Ambiguous cases must not be auto-renamed.

Examples of ambiguous cases:

- a legacy node with no provenance metadata and a specific non-generic name
- a node whose current name differs from the inferred generated suggestion
- a node whose source cannot be reconstructed confidently from tags, node ID,
  merged proposal history, or event history

### Step 6: Surface suggestions for manual review

For ambiguous legacy nodes:

- preserve the current `name`
- store naming metadata if possible
- keep the latest suggestion in `generated_name`
- expose the suggestion in the detail and edit flows

### Step 7: Lock in regeneration behavior

After backfill:

- future spec and discovery passes should update existing generated components
  by `origin_key`
- manual names should remain stable until the user explicitly adopts a new
  suggestion

## Risks, Dependencies, And Rollout Order

### Main risks

- changing dedupe from name-based to origin-based matching may reveal existing
  duplicates that were previously hidden
- legacy discovery nodes may not have enough persisted origin detail for
  confident automatic backfill
- if the phase appends subtype-like words to names too aggressively, Phase 02
  display labels and Phase 03 instance names may become redundant
- the synthetic factory-output node may still feel product-awkward even after
  it gets a better name

### Dependencies

- Phase 02 taxonomy work must remain the source of subtype display labels so
  this phase does not reintroduce `Module` through UI copy
- Phase 00 project identity work improves fallback naming quality because
  project names and slugs become more reliable context
- the existing blueprint history and event log features should be used for
  migration safety and rollback confidence

### Recommended rollout order

1. Land schema support and the shared naming engine.
2. Update spec emission, discovery scanning, and origin-based dedupe.
3. Add manual ownership handling in create, edit, and accept flows.
4. Expose provenance and suggested-name UI in discovery and detail views.
5. Run preview-only backfill.
6. Apply safe renames and review ambiguous cases manually.
7. Update docs and tests to lock the new rules in place.

## Unresolved Questions

- Should generated names prefer subject-only forms like `Authentication` when
  the component subtype is already visible elsewhere, or should names normally
  include a role suffix like `Authentication Service`?
- Should the discovery accept flow support inline rename in the first rollout,
  or is accept-then-edit sufficient for now?
- Is the synthetic factory-output component still a valid product concept, or
  should it eventually become build evidence instead of a component node?
- How aggressive should automatic legacy backfill be for names that are weak
  but not obviously placeholders, such as `Configuration` or `Admin Panel`?

## Done When

Phase 03 is complete when the following statements are true:

- no new generated component name falls back to `Unknown Module`, `X Module`,
  `Api`, `Core`, `Web`, or `Factory Output`
- every generated component has stable naming provenance and an `origin_key`
- manual component names survive spec and discovery reruns
- discovery dedupe and merge behavior no longer depends on the current display
  name
- legacy generic component names have a reviewed backfill plan, with safe
  renames applied and ambiguous cases preserved for manual review
- tests fail if generic fallback names or unsafe overwrite behavior return
