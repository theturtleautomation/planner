## Purpose

Define the current **UI-reset family contract** beneath the project-picture
experience layer.

This surface preserves the original route-by-route reset program as completed
historical lineage, but it is no longer an implicitly live execution backlog.
Its job now is to state what UI reset owns, what it does not own, and how any
future follow-on route work should be opened.

## Current UI-Reset Family Contract

UI reset is the experience-layer family that owns:
- route posture,
- dominant-surface hierarchy,
- supporting-context visibility,
- and reveal-model discipline for route families.

UI reset answers questions like:
- what each route should make primary,
- what context should remain visible versus move into a reveal surface,
- how routes should communicate progress/status clearly,
- and how page models should feel calmer and more product-directed without
  hiding real operational truth.

## Boundary Model

### UI reset owns
- route-level posture and focal-surface hierarchy,
- reveal-model discipline for drawers, inspectors, sheets, and overlays,
- route-specific information-density and supporting-context decisions,
- and the product-model clarity of page families such as directories,
  workspaces, review consoles, queues, and operations surfaces.

### UI reset does not own
- the shared cross-product experience grammar; that belongs to the design
  system family,
- or the specialized knowledge-library domain thesis; that belongs to the
  knowledge family,
- or deeper structural concerns such as hidden truth-model, recoverability, and
  overlay/reorientation modeling.

## Relationship To Sibling Families

### Design system remains a sibling family

Design system defines the shared experience grammar used across routes:
semantic surfaces, visual language, shared interaction rules, and reusable
cross-route patterns.

UI reset consumes that grammar, but does not replace or own it.

### Knowledge library remains a sibling family

Knowledge library defines the specialized scoped-workspace thesis for knowledge
inventory, rationale, and project-first knowledge context.

UI reset can inform the posture of knowledge routes historically, but
knowledge-workspace-specific future planning belongs to the knowledge family,
not to UI reset as an open-ended parent surface.

## What Future UI-Reset Work Should Look Like

Future UI-reset work should open only when a fresh route-posture gap is found,
for example:
- a route still lacks a clear dominant surface,
- supporting context still competes with primary work,
- a reveal model remains muddy or overexposed,
- or a new route family needs bounded posture planning.

Future work should **not** reopen the original queue by default.

Instead, open a new follow-on spec only when:
- the problem is genuinely about route posture / hierarchy / reveal discipline,
- the problem is not primarily shared-grammar work for design system,
- and the problem is not primarily domain-thesis work for knowledge library or
  another sibling family.
