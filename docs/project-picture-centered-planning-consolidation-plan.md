# Project-Picture-Centered Planning Consolidation Plan

**Status:** Implemented  
**Date:** 2026-04-08  
**Parent:** [Socratic Project Picture And Convergence Workspace Spec](/home/thetu/planner/docs/socratic-project-picture-and-convergence-workspace-spec.md)

## Purpose

Make the **Socratic project picture** the canonical planning center for the
next product-shaping layer, instead of leaving several adjacent families as
competing top-level centers.

This plan does not replace the project-picture spec. It is the bounded
consolidation container that reorganizes the planning lanes around that center.

## Why This Exists

Recent work made the project picture feel like the most coherent product
direction, but the planning tree still leaves these as parallel top-level
surfaces:

- Planner design system
- Planner UI reset
- Knowledge library

Those are no longer best understood as independent centers. They are now
better treated as subordinate to the project-picture-centered product model.

At the same time, the deeper Socratic concerns remain important:

- hidden truth-model / blueprint relationship
- whole-project recoverability beyond same-route shell
- richer overlay / reorientation model

Those concerns should remain active, but as a separate child layer rather than
being merged into the same immediate sub-plan as design/UI/knowledge.

## Consolidation Shape

### Center

- **Socratic project picture**

### Child Layer 1 — Experience consolidation

These plans should be reframed beneath the project-picture center:

- [Planner Design System Command Center Plan](/home/thetu/planner/docs/planner-design-system-command-center-plan.md)
- [Planner UI Reset Route-By-Route Spec Queue](/home/thetu/planner/docs/planner-ui-reset-route-by-route-spec-queue.md)
- [Knowledge Library Project Scope Plan](/home/thetu/planner/docs/knowledge-library-project-scope-plan.md)

### Child Layer 2 — Structural concerns

These remain active and important, but should be treated as a separate child
layer:

- hidden truth-model / blueprint relationship
- whole-project recoverability beyond same-route shell
- richer overlay / reorientation model

## Explicit Non-Goals

- do not make planner-ledger the design center
- do not pull import-existing-project into this pass
- do not collapse all child layers into one giant merged super-plan

## Outcome Target

After this consolidation lands:

- the project picture is the unmistakable planning center
- design/UI/knowledge no longer compete as peer top-level planning centers
- the deeper Socratic concern family remains active and attached, but in its
  own layer

## Next Move

Use this container to synchronize the affected plans and ledger structure, then
decide whether the new center needs another planning pass or an implementation
slice.
